===========================================
二、技术栈与规范铁律
===========================================

【后端技术栈】
- 框架：FastAPI + Uvicorn
- 推理引擎：LangGraph (Plan-and-Execute 模式)
- 状态管理：SQLite Checkpointer (持久化 AgentState)
- LLM 调用：OpenAI SDK (兼容多种 LLM API)
- 数据处理：Pandas + NumPy
- 异步支持：aiohttp + httpx + aiomysql + asyncpg

【前端技术栈】
- 纯静态 HTML/CSS/JavaScript
- SSE (Server-Sent Events) 流式通信

【核心架构】
- LangGraph 状态机：Intent → Retrieval → Planner → Executor → Analyzer
- 数据隔离：所有 API 查询结果存入 data_context 字典，避免 Prompt 污染
- 人工审批网关：interrupt_before=['Executor'] 实现审批挂起
- 权限控制：RBAC 基于用户角色和 API 授权

【禁止事项】🚫
- 禁止引入同类替代框架（如 LangChain、Haystack）
- 禁止修改 AgentState 顶层结构
- 禁止破坏 LangGraph 主流程
- 禁止在 Prompt 中拼接大量数据（必须使用 data_context）
- 禁止擅自删除审批模块代码（仅调整触发条件）

十一、F-03 功能完成记录
===========================================

【完成时间】2026-03-27
【完成智能体】编码智能体 (Coding Agent)
【功能ID】F-03
【功能描述】修复Retrieval到Planner的API元数据传递缺失问题

【问题分析】
1. Qdrant 使用 file:// 存储模式导致并发访问冲突
   - 服务器启动时锁定存储文件
   - 查询时无法访问向量库，导致检索返回 0 个 API
2. 服务器启动时未构建 API 向量索引
3. Endpoint 结构使用 params_mapping，但 Planner 期望 params 字段

【实现方案】
1. 修改 settings.py：
   - 将 QDRANT_URL 从 "file://./data/qdrant" 改为 ":memory:"
   - 解决并发访问问题，每个进程独立内存存储

2. 修改 main.py：
   - 在 lifespan 启动阶段添加 API 向量索引构建
   - 遍历所有 API 并调用 build_index_for_api()
   - 确保服务器启动后索引可用

3. 修改 api_retrieval_service.py：
   - 在 _enrich_with_full_metadata() 中规范化 endpoint 结构
   - 将 params_mapping 复制到 params 字段
   - 确保 Planner 能正确读取参数信息

【测试结果】
✅ API 向量索引成功构建（5 个 API）
✅ 检索服务返回完整 API 元数据
✅ API ID、名称、endpoints 正确传递
✅ Endpoint params 从 0 增加到 5 个参数

- backend/app/services/api_retrieval_service.py
