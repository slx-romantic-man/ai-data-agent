===========================================
一、基础设施清单
===========================================

本轮创建的三大"基建事实载体"：

1. init.sh (根目录)
   - 用途：防御性环境启动脚本
   - 功能：依赖检查、虚拟环境、端口检查、健康检查
   - 使用：bash init.sh (任何智能体接手时首先运行)

2. feature_list.json (根目录)
   - 用途：项目功能进度的唯一真相来源
   - 格式：严格 JSON (禁止 Markdown)
   - 结构：11个原子化功能卡片 (F-09 到 F-19)
   - 状态：每个功能带 passes 布尔值标记

3. claude-progress.txt (本文件，根目录)
   - 用途：智能体交接日志，人类/机器双读
   - 内容：技术栈、工作纪律、进度记录、失败模式警告

===========================================
二、技术栈与铁律
===========================================

【后端技术栈】
- 框架：FastAPI (REST API)
- 工作流引擎：LangGraph (Agent 编排)
- ORM：SQLAlchemy
- LLM 客户端：OpenAI SDK
- 数据库：SQLite (checkpoints.db)
- 会话存储：JSON 文件 (sessions.json)

【前端技术栈】
- 纯静态：HTML + CSS + JavaScript
- 禁止引入：React、Vue、Angular 等框架

【Agent 架构】
Intent → Retrieval → Planner → Executor → Analyzer

【核心节点】
- Intent Node: 意图澄清与实体提取
- Retrieval Node: API 向量召回与 LLM 精排
- Planner Node: 全局规划生成 JSON 执行计划
- Executor Node: 工具路由与执行
- Analyzer Node: 终局分析与答案生成

【工具集】
- api_fetch_tool.py: 外部 API 调用
- sql_query_tool.py: SQL 查询
- python_exec_tool.py: Python 代码执行
- analysis_tool.py: 数据分析
- export_tool.py: Excel 导出

【铁律】
🚫 禁止更换技术栈 (如用 Django 替换 FastAPI)
🚫 禁止引入同类替代库 (如同时用 requests 和 httpx)
🚫 禁止修改 Agent 架构的五节点流程
🚫 禁止在前端引入 npm 框架

===========================================
三、标准启动序列
===========================================

任何接替工作的智能体必须按以下顺序执行：

1. pwd
   → 确认当前工作目录

2. cat claude-progress.txt
   → 读取本文件，了解项目状态与工作纪律

3. cat feature_list.json
   → 读取功能清单，确定下一个待完成功能

4. bash init.sh
   → 启动环境，进入可测试状态

5. 选择功能
   → 从 feature_list.json 中选择 priority 最高且 passes=false 的功能

6. 实现功能
   → 只做一个功能，不贪多

7. 更新状态
   → 修改 feature_list.json 中对应功能的 passes=true
   → 在本文件末尾追加进度记录

8. Git 提交
   → git add .
   → git commit -m "feat/fix/test: 功能描述"

===========================================
四、工作纪律
===========================================

【单功能原则】
- 每次会话只做一个功能
- 按 priority 从小到大选择 (priority 越小越优先)
- 同优先级按 id 顺序选择

【验收标准】
- 不以代码存在为准
- 不以接口返回 200 为准
- 必须以 feature_list.json 中的 steps 全部通过为准
- 对于用户交互功能，必须等待 Agent 给出最终回答才算通过

【状态更新】
- 完成功能后立即更新 feature_list.json
- 在本文件末尾追加进度记录
- Git 提交必须包含清晰的 commit message

【失败模式警告】
🚫 禁止一次性实现多个功能 (会导致测试不充分、回滚困难)
🚫 禁止仅凭代码存在就判断项目完成 (必须以 feature_list.json 为准)
🚫 禁止跳过 feature_list.json 更新 (会导致进度混乱)
🚫 禁止修改已完成功能的代码 (除非该功能被标记为 passes=false)
