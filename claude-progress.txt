================================================================================
  二、技术栈与规范铁律 (🚫 禁止擅自引入同类替代品)
================================================================================

  [后端]
  - 语言:       Python 3.8+
  - 框架:       FastAPI (uvicorn)
  - AI 编排:    LangGraph (StateGraph + AsyncSqliteSaver)
  - LLM 客户端: 自研 BaseLLMClient (OpenAI/Aliyun/Anthropic compatible)
  - HTTP 客户端: httpx (异步)
  - 数据库:     SQLAlchemy 2.0 (async) + aiosqlite
  - 向量搜索:   Qdrant (qdrant-client) + sentence-transformers
  - 数据处理:   Pandas + NumPy
  - 缓存:       自研 LLMResponseCache (app/utils/llm_cache.py)
  - 导出:       openpyxl (Excel)

  [前端]
  - 类型:       纯静态 HTML/CSS/JS (无框架, 无构建步骤)
  - 入口:       frontend/index.html
  - 通信:       SSE (Server-Sent Events) + REST API

  [基础设施]
  - 日志:       自研 app/utils/logger.py
  - 配置:       pydantic-settings (.env 驱动)
  - 部署:       Docker + docker-compose + nginx
  - 状态持久化: SQLite (WAL 模式)

  🚫 禁止引入: Flask, Django, LangChain (非 LangGraph), axios,
               React, Vue, Angular, Webpack, Vite, TailwindCSS,
               或任何与上述栈功能重叠的替代品。

================================================================================
  四、核心工作流 (LangGraph 节点链)
================================================================================

  START → Retrieval → IntentPlanner → Executor (循环) → Analyzer → END

  性能瓶颈:
  - IntentPlanner LLM 调用: ~42s (prompt 过长, max_tokens=4096)
  - Analyzer LLM 调用:      ~85s (7段式报告, prompt 臃肿)
  - 总计 LLM 占比:          96%

================================================================================
  六、工作纪律 (🚨 后续每个 AI Agent 必须遵守)
================================================================================

  1. 每次会话只做一个功能
     - 打开 feature_list.json
     - 找到 priority 最小且 passes=false 的功能
     - 只做那一个

  2. 完成后更新状态
     - feature_list.json 中该功能的 passes 改为 true
     - 在本文件 "七、进度日志" 追加一条记录

  3. Git 提交
     - 提交格式: feat(perf): F-XX 功能简述
     - 例如: feat(perf): F-01 skip analyzer for simple queries

  4. 失败模式警告
     🚫 禁止一次性实现多个功能
     🚫 禁止仅凭代码存在就判断功能完成，必须以 feature_list.json 为唯一真相
     🚫 禁止引入与现有技术栈重叠的新依赖

  [2026-04-14] Session #15 — 编码智能体 (F-13 修复)
  ✅ F-13: 修复打字机效果不生效的根因
     - 根因: Vue 3 的 ref([]) 数组中的对象被 Proxy 包裹。当 setInterval 修改对象的
       content 属性时，Vue 能检测到 state 变化，但 v-for + v-html 组合不会触发 DOM 重渲染。
       setInterval 在 18ms 间隔的修改操作被 Vue 的 scheduler 批量处理后永远不会 flush 到 DOM。
     - 修复方案: 将 setInterval 替换为 async/nextTick 循环，并在每次修改后使用
       messages.value.splice(idx, 1, { ...assistantMsg }) 替换数组元素，强制 Vue 重新渲染 v-for。
     - 修改文件:
       · frontend/js/features/chat.js: runAnswerTypewriter() 使用 nextTick + splice 替代 setInterval

  ✅ F-32: 诊断 yield from / yield 在 async generator 上下文中的行为

  📝 修改文件:
    - backend/test_f32_diagnostic.py (新增诊断脚本)

  🔬 诊断结论 (Python 3.13.3):
    1. `yield sync_gen()` 在 async def 中：
       → 产出 generator 对象本身（不是其内部值）
       → async for 只收到 generator object，无 __aiter__ 方法 → TypeError
    2. `yield from sync_gen()` 在 async def 中：
       → 语法错误（SyntaxError: 'yield from' cannot be used in async function）
    3. 正确做法：
       → 在 async generator 内手动迭代 sync generator:
         for chunk in sync_gen(text):
             await asyncio.sleep(0)  # 可选的 yield 点
             yield chunk
