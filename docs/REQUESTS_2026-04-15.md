## 一、动态智能推荐问题（F-22）

### 1.1 需求背景

现有实现中，推荐问题只在**首次登录后的智能对话界面**显示一次，之后不再刷新。

**用户反馈的具体问题**：
- 用户被管理员授予新 API 权限后，刷新智能对话页面，推荐问题仍然是旧的一组，没有根据新权限动态更新
- 管理员给员工赋予了新 API，员工自己看不到新的推荐问题
- 管理员给自己添加了新 API，管理员自己刷新页面也看不到新推荐
- 未被授予任何权限的普通员工账号，智能对话页显示了与管理员相同的一组推荐问题（权限隔离失效）

**根本原因**：推荐问题在首次加载后被缓存，没有建立「权限变更 → 推荐刷新」的联动机制。

---

### 1.2 功能定义

#### 1.2.1 触发时机

以下任意一个事件发生时，必须重新获取并渲染推荐问题：

| 触发时机 | 说明 |
|---|---|
| 用户点击「新对话」按钮 | 每次点击都视为一次全新的对话上下文，必须刷新推荐 |
| 用户切换到「智能对话」Tab（页面从后台恢复） | 通过 `visibilitychange` 事件检测，页面从隐藏状态恢复时强制刷新 |
| 用户登录成功（首次或重新登录） | 登录完成后立即获取最新权限对应的推荐 |
| 管理员在「权限概览」页面操作了权限变更 | 变更提交后，管理员自己页面的推荐应立即更新（无需刷新） |

#### 1.2.2 数据来源（必须实时，不允许缓存）

推荐问题的数据源是 `/api/v1/chat/suggestions` 接口。

该接口接收 `user_id`（从登录态获取），返回该用户**当前实际拥有**的 API 权限所对应的推荐问题列表。

**后端必须实现的逻辑**：
1. 根据 `user_id` 查询该用户被授予了哪些 API（查 `api_permissions` 表，过滤 `is_active=true`）
2. 根据 API 的 `config_id` + `name` 生成对应的问题推荐
3. **每次请求实时查询**，不允许在服务端缓存推荐结果（缓存应发生在用户级别，且不超过 5 分钟）
4. **管理员账号特殊处理**：管理员拥有所有 API 的访问权限（`role='admin'`），其推荐应为所有 API 的合集

#### 1.2.3 前端必须实现的逻辑

1. **「新对话」按钮点击**：
   - 点击后，调用 `/api/v1/chat/suggestions` 获取最新推荐
   - 渲染到对话界面的推荐问题区域
   - 记录本次请求的 `user_id` + `timestamp`，用于对比

2. **页面可见性变化（`visibilitychange`）**：
   - 当 `document.visibilityState` 从 `hidden` 变为 `visible` 时，重新获取推荐
   - 防止用户在另一个标签页授予权限后，当前页不刷新

3. **权限变更通知（Admin 场景）**：
   - 管理员在「权限概览」操作权限后，通过 WebSocket 或轮询通知当前在线的管理员页面刷新推荐（**可选方案 A**）
   - 或者管理员操作权限后，只要当前页面检测到 `visibilitychange`，立刻刷新（**默认方案 B**，实现简单）

4. **防重复渲染**：
   - 推荐问题区域应有唯一标识（如 `data-section="suggestions"`）
   - 每次刷新前先清空旧内容，避免残留

#### 1.2.4 权限隔离要求

| 场景 | 推荐问题内容 |
|---|---|
| 普通员工（0 个 API 权限） | 显示「暂无可用的分析能力，请联系管理员开通」，**不能显示任何具体 API 对应的问题** |
| 普通员工（3 个 API 权限） | 仅显示这 3 个 API 对应的推荐问题 |
| 管理员账号 | 显示所有 API 的推荐问题（完整合集） |
| 同一员工被管理员更新了权限（从 3 个变成 5 个） | 立即刷新后显示新的 5 个推荐 |

---

### 1.3 接口规格

**请求**
```
GET /api/v1/chat/suggestions
Headers:
  Authorization: Bearer <token>
```

**响应**
```json
{
  "suggestions": [
    {
      "question": "查询北京的天气",
      "api_name": "weather_api",
      "icon": "🌤️"
    },
    {
      "question": "查询 AAPL 股票价格",
      "api_name": "alpha_vantage_stock",
      "icon": "📈"
    }
  ],
  "total": 2,
  "generated_at": "2026-04-15T16:45:00Z"
}
```

**后端实现位置**: `backend/app/api/v1/chat.py` 的 `get_suggestions` 函数

## 二、历史记录增强（历史对话查看体验优化）

### 2.1 需求背景

用户通过「历史记录」列表点击进入某次对话后，再点返回「历史记录」页面时存在两个体验问题：
1. 列表总是回到最顶部，最新一条在最上，用户每次都要重新滚动找到之前的位置
2. 被选中的那条历史记录没有高亮标记，用户无法确认自己看的是哪一条

此外，部分历史对话的详情页没有展示 Agent 的思考过程（reasoning log），且无法导出当时查询到的结构化数据。

---

### 2.2 功能定义

#### 2.2.1 历史记录列表：保留滚动位置

**现状**：点击某条历史记录 → 跳转到「智能对话」页 → 点返回 → 列表刷新到顶部

**目标**：点击某条历史记录 → 跳转到「智能对话」页 → 点返回 → 列表保持离开时的滚动位置，且被选中的那条历史记录保持 `.selected` 高亮状态

**前端实现方案**：

```javascript
// 在 history 相关 JS 中维护一个 sessionStorage key
const HISTORY_SCROLL_KEY = 'history_list_scroll_top';
const HISTORY_SELECTED_KEY = 'history_selected_session_id';

// 离开历史列表时保存状态
window.addEventListener('beforeunload', () => {
  const scrollTop = document.querySelector('.history-list')?.scrollTop || 0;
  const selectedId = document.querySelector('.history-item.selected')?.dataset.sessionId;
  sessionStorage.setItem(HISTORY_SCROLL_KEY, scrollTop);
  sessionStorage.setItem(HISTORY_SELECTED_KEY, selectedId || '');
});

// 返回历史列表时恢复状态
onMount(() => {
  const scrollTop = parseInt(sessionStorage.getItem(HISTORY_SCROLL_KEY) || '0');
  const selectedId = sessionStorage.getItem(HISTORY_SELECTED_KEY) || '';
  // 恢复滚动位置
  const listEl = document.querySelector('.history-list');
  if (listEl) listEl.scrollTop = scrollTop;
  // 恢复选中状态
  if (selectedId) {
    const el = document.querySelector(`.history-item[data-session-id="${selectedId}"]`);
    if (el) el.classList.add('selected');
  }
});
```

**关键要求**：
- 使用 `sessionStorage`（不是 `localStorage`），因为这是标签页内的状态传递，关闭标签页应清除
- 列表回到历史页面后，选中的那条应有明显的 `.selected` 样式（背景高亮或左边框标记）

#### 2.2.2 所有历史对话：必须展示 Agent 思考过程

**问题**：部分历史对话详情页没有展示 reasoning log（`thought` 步骤），用户只能看到最终答案，无法回溯 Agent 的推理过程。

**分析**：通过 `/api/v1/chat/sessions/{session_id}` 获取的 session 数据中，如果包含 `reasoning_log` 或 `messages` 中的 `thought` 类型消息，前端必须渲染出来。

**前端要求**：
1. 读取 session 数据中的所有 `messages`
2. 筛选 `role='assistant' 且 content 中包含推理步骤类型`的消息
3. 在对话详情页面，将推理过程以**折叠面板**（默认展开）的形式展示在最终答案上方
4. 面板标题：「Agent 思考过程」，点击可展开/收起
5. 每个思考步骤显示 `step` 编号和对应的 `content`

**展示格式参考**：
```
[展开] Agent 思考过程
  步骤 1: 我先检索当前可用的数据源，已找到 3 个 API...
  步骤 2: 用户查询苹果近期股价，意图明确指向金融数据...
  步骤 3: 调用 alpha_vantage_stock API 获取数据...
  步骤 4: 数据已经准备好了，我正在整理结果...
```

#### 2.2.3 历史对话详情：支持导出数据

**需求**：在对话详情页，每个带有 `data.rows` 的 assistant 消息旁边，需要显示一个「导出数据」按钮。

**行为**：
1. 点击按钮，导出该条消息对应的 `data.rows` 为 `.xlsx` 文件
2. 文件名为：`{session_id}_{timestamp}.xlsx`
3. 如果该条消息没有 `data.rows`（如澄清类对话），按钮不显示

**前端实现**：参考现有的 `/api/v1/export/{session_id}.xlsx` 接口，复用导出逻辑。

#### 2.2.4 部分历史对话没有思考过程的排查

**根因分析**：Session 数据中存储的 `messages` 是否包含 `thought` 消息，取决于 `stream_chat` 端点发送给前端的事件是否被正确保存。

检查点：
1. `/api/v1/chat/sessions/{session_id}` 返回的历史消息中，是否包含 `type='thought'` 的消息？
2. 如果历史消息中**有** `thought`，前端是否渲染了？——可能是前端读取时过滤掉了
3. 如果历史消息中**没有** `thought`（只有最终 answer），说明 `stream_chat` 端点没有把 `thought` 事件存入 session，需要修复存储逻辑

**建议**：先通过 curl 调用 `/api/v1/chat/sessions/{session_id}` 验证返回的 JSON 结构，确认数据层是否有 `thought` 消息，再决定前端还是后端修。

---

### 2.3 改动范围

| 文件 | 改动说明 |
|---|---|
| `frontend/js/features/history.js`（或相关） | 1. `beforeunload` 保存滚动位置和选中 ID 到 `sessionStorage`<br>2. `onMount` 恢复滚动位置和选中状态<br>3. 列表项增加 `.selected` 高亮样式 |
| `frontend/js/app-template.js` | 4. 对话详情页中渲染 Agent 思考过程（折叠面板）<br>5. 带有 `data.rows` 的消息旁增加「导出数据」按钮 |
| `backend/app/api/v1/chat.py` | 6. 确认 session 存储时包含 `thought` 类型的消息内容 |

### 2.4 验收标准

- [ ] 点击历史记录进入对话，再返回历史列表，列表保持在离开时的滚动位置

## 五、注意事项

1. **F-22 权限隔离是 P0**：未授权员工看到与管理员相同的推荐问题是严重的权限漏洞，上线前必须修。
2. **历史记录推荐问题缓存**：不要在 `localStorage` 或 `sessionStorage` 中缓存推荐结果，每次点击「新对话」必须实时调用 API。
3. **Admin 自更新**：管理员操作权限后，他自己的推荐问题应立即刷新（无需重新登录），注意 `role='admin'` 的判断逻辑。
4. **历史思考过程**：先通过 API 调试工具（如 curl）确认 `/api/v1/chat/sessions/{session_id}` 返回的数据结构，再决定前端如何渲染。
